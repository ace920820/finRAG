# FinRAG 开发过程复盘：从 Mock RAG 到金融研究型 RAG Demo

## 0. 总体目标

**Situation / 背景**

项目初始目标是构建一个金融领域 RAG Agent MVP，用于面试/演示场景。它不是通用聊天机器人，而是面向金融研究人员、投研分析师的“可追溯、可引用、可解释”的问答系统。

核心要求包括：

- 能基于本地金融文档回答问题。
- 能展示检索过程：Query 改写、BM25 召回、向量召回、Rerank。
- 能给出 Markdown 格式的结构化回答。
- 能展示引用来源，支持点击 citation 查看证据。
- 能支持 PDF 文档导入、索引构建、知识库维护。
- 后续要便于前后端联调，而不是把重点放在前端视觉设计上。

**Task / 任务**

我们需要把系统从最早的 mock-data 原型，逐步强化成一个可演示的真实金融文档 RAG 系统，并解决金融文档中特有的问题：

- PDF 文档导入。
- 表格数据解析。
- 财报最新数据识别。
- 数字型问题精准回答。
- 引用与证据链展示。
- 知识库管理与重新索引。
- 模型 provider 可配置。
- 前后端单应用集成。

# 1. 第一阶段：搭建 Mock-data MVP 与基本 RAG 骨架

## OODA / STAR

### Observe / Situation

最开始我们只有需求文档，没有可运行的后端服务，也没有稳定的 RAG 工作流。前端设计会由其他团队提供，所以我们的重点应放在：

- 后端能力。
- API 契约。
- 检索链路。
- SSE 流式输出。
- 前后端后续联调准备。

### Orient / 判断

我们判断初期不应过早投入真实 PDF pipeline，否则会拖慢系统骨架搭建。更合理的路径是：

1. 先用 mock 数据建立完整闭环。
2. 把后端 API、检索流程、SSE 事件、前端数据结构跑通。
3. 再逐步替换成真实文档与真实索引。

### Decide / Action

因此 v1.0 做了这些基础工作：

- 建立 FastAPI 后端。
- 建立基本 RAG 工作流：
  - Query 处理。
  - BM25 检索。
  - 向量检索。
  - Rerank。
  - Answer generation。
- 建立 SSE 查询接口。
- 前端能够展示：
  - 用户问题。
  - 阶段进度。
  - 召回结果。
  - Rerank Top 5。
  - Markdown 回答。
  - Citation。
- 增加基础测试，保证 API 行为稳定。

### Result

结果是系统具备了第一个可演示闭环：即使数据是 mock，也已经能展示完整 RAG 流程。这为后续真实语料、PDF pipeline、知识库管理提供了稳定底座。

# 2. 第二阶段：模型 Provider 抽象与阿里云百炼接入

## Observe

早期系统不能绑定本地模型。你明确提出：

- Embedding 不使用本地模型。
- Rerank 不使用本地模型。
- 文本模型使用阿里云百炼 Qwen Plus。
- Embedding、Rerank、Text model 都要可以通过配置统一切换。

### Orient

我们判断这里最重要的不是“接一个 API”，而是建立 provider 抽象：

- 避免把百炼写死到业务逻辑里。
- 测试环境继续使用 mock provider，避免 CI 或本地测试依赖真实 API Key。
- `.env` 中预留必要变量，让部署时只改配置，不改代码。

### Decide / Action

我们做了：

- 增加统一模型配置项：
  - `FINRAG_TEXT_PROVIDER`
  - `FINRAG_EMBEDDING_PROVIDER`
  - `FINRAG_RERANK_PROVIDER`
  - `FINRAG_TEXT_MODEL`
  - `FINRAG_EMBEDDING_MODEL`
  - `FINRAG_RERANK_MODEL`
  - `FINRAG_MODEL_API_KEY_BAILIAN`
  - `FINRAG_MODEL_API_KEY_SILICON`
  - `FINRAG_MODEL_BASE_URL_*`
- 接入百炼 Qwen Plus 文本模型。
- 后续又增加 SiliconFlow provider，使 embedding/rerank 可以切换到硅基流动：
  - `BAAI/bge-m3`
  - `BAAI/bge-reranker-v2-m3`
- 在模型切换后重建索引，避免向量维度和旧索引不一致。

### Result

系统从“单 provider Demo”强化为“模型服务可替换的 RAG 系统”。后续部署到服务器时，可以通过 `.env` 选择百炼/硅基流动，而不用改业务代码。

# 3. 第三阶段：真实 PDF 导入 Pipeline

## Observe

Mock 数据通过冒烟测试后，核心问题变成：系统没有真实资料导入能力。你提供了已有的 `pdf2md` 项目，它原本能从 PDF 文本层提取内容并结构化处理。

我们观察到如果没有导入 pipeline，系统会有几个根本限制：

- 无法处理用户自己的 PDF。
- 向量库来源不透明。
- 数据更新依赖手工改 JSON。
- 知识库维护无法闭环。

### Orient

我们判断应该复用 `pdf2md`，但要把职责拆清楚：

- `pdf2md` 负责 PDF → raw Markdown / raw artifacts。
- FinRAG backend 负责 raw Markdown → documents/chunks/index。
- 检索运行时继续读取 `processed` 和 `index` 目录。

这样可以避免把 PDF 解析、RAG schema、向量索引全部耦合在一个脚本里。

### Decide / Action

v1.1 中我们增加了两个核心 phase：

#### Phase 6：PDF Extraction Adapter

- 给 `pdf2md` 增加 `--profile finrag`。
- 支持通用金融 PDF，而不再绑定原项目的特定文件命名/业务假设。
- 输出 raw Markdown、frontmatter、manifest。
- 保留源文件路径、标题、页数、hash、处理状态。
- 支持 idempotent skip、`--force`、单文件失败隔离。

#### Phase 7：FinRAG Corpus Import And Index Build

- 增加 backend 侧 import pipeline。
- 解析 raw `.md` / `.txt`。
- 生成：
  - `documents.json`
  - `chunks.json`
  - index 文件
- 增加 `backend/scripts/import_corpus.py`。
- 增加 `backend/scripts/build_index.py`。
- 导入真实 40 篇 PDF：
  - 宁德时代 10 篇。
  - 贵州茅台 10 篇。
  - NVIDIA 10 篇。
  - 台积电 10 篇。
- 构建真实语料索引。

### Result

系统从 mock-only 进入真实文档 RAG 阶段。GSD 文档记录的结果是：

- 真实 PDF：40 个。
- 导入文档：40 篇。
- 初始文本 chunks：9303 个。
- 支持从真实文档库进行检索与问答。

这一步是系统性质的转折点：从“演示 RAG 流程”变成了“可以维护真实金融知识库”。

# 4. 第四阶段：前端证据链与检索过程可视化强化

## Observe

真实语料导入后，新的问题是：用户不仅需要答案，还要知道答案从哪里来、当前回答用了哪些证据。

早期前端右侧证据面板容易出现两个问题：

- 只展示“最新一次检索”的证据。
- 回看旧回答时，证据可能和当前选中回答不对应。

### Orient

我们判断金融研究场景里，证据链比普通聊天更重要。因此不能只做全局状态，而应该把每轮回答的检索快照存下来。

### Decide / Action

Phase 8-10 做了这些强化：

#### Phase 8：真实语料入口

- 左侧示例问题改成真实 corpus 相关问题。
- 文档库条目可点击。
- 增加 `/api/documents/{doc_id}/view`。
- 文档列表展示 source metadata。

#### Phase 9：检索侧栏交互优化

- BM25、Vector、Rerank 面板可独立折叠。
- Rerank item 可以展开查看完整证据文本。
- 统一 score 格式和布局。
- 增加空状态/全部折叠状态。

#### Phase 10：Per-turn Evidence Traceability

- 每个 assistant message 存储自己的 retrieval snapshot。
- SSE 过程中记录：
  - BM25 docs。
  - Vector docs。
  - Rerank docs。
  - citations。
- 点击某个历史回答时，右侧显示该回答对应证据，而不是最新全局证据。
- 点击 citation 时能定位到对应证据。

### Result

系统从“能回答”增强为“能解释为什么这么回答”。这是金融 RAG Demo 的关键展示价值：不仅有答案，还有检索过程和引用链路。

# 6. 第六阶段：严重数据丢失 Bug 排查与导入策略修正

## Observe

你在前端上传一个 PDF 并确认导入后，发现：

- 原本 42 个文档和 chunks 被清零。
- RAG 问答无法检索。
- 后续恢复后又出现文档数变成 68 的异常。

### Orient

我们判断这是知识库导入策略的问题，而不是前端展示问题。核心风险是：

- 新导入的临时数据覆盖了已有 processed corpus。
- 空导入结果也可能写回正式 corpus。
- 重复导入导致文档去重不严格。
- 索引与 processed 数据不一致。

### Decide / Action

我们做了几类修复：

- 禁止 0 文档导入覆盖现有 corpus。
- API 导入改成 merge 流程，而不是直接覆盖。
- 上传导入使用临时目录，成功后再合并。
- 恢复 40 篇预期 corpus。
- 重建 raw / processed / index。
- 增加回归测试：
  - PDF-only incremental import。
  - Markdown incremental import。
  - 防止空导入覆盖。
- 对 source document 重新导入，恢复 40 篇、9303 chunks 的基础数据。

### Result

知识库导入从危险的“覆盖式导入”调整为更安全的“增量合并式导入”。系统恢复了：

- 40 篇预期文档。
- 9303 文本 chunks。
- RAG 检索能力。

这一步解决的是数据治理层面的稳定性问题。

# 7. 第七阶段：表格解析能力强化

## Observe

金融 PDF 中大量关键信息不在普通段落里，而在表格里，比如：

- 营收。
- 净利润。
- 分业务收入。
- 财务季度数据。
- 年度对比数据。

早期系统对 PDF 表格的处理方式主要是文本层抽取，也就是把表格当普通文本处理。这会导致：

- 行列结构丢失。
- 指标与期间对应关系弱。
- 问“最新季度营收是多少”时容易拿错年份/季度。
- 检索命中的 chunk 可能包含数字，但模型不一定能理解表格结构。

### Orient

我们判断金融 RAG 不能只依赖“多切 chunk + 交给模型理解”。尤其是财务数字问题，应该把表格提升为一等证据对象。

因此我们新增 Phase 14-16，专门做 table-aware RAG。

---

## 7.1 Phase 14：Table-Aware PDF Extraction

### Decide / Action

我们扩展 `pdf2md --profile finrag`：

- 使用 `pdfplumber` 提取表格。
- 保留 PyMuPDF 原本文本抽取。
- 输出结构化表格 artifact：
  - JSON。
  - CSV。
  - manifest。
- 每个 table artifact 保留：
  - page。
  - inferred title/caption。
  - headers。
  - rows。
  - row/column count。
  - source PDF metadata。
  - Markdown table rendering。
- 表格提取失败时降级为 text-only，不影响已有 pipeline。
- 对 40 篇 PDF 重新处理。

### Result

系统开始保留 PDF 表格结构，而不只是文本层内容。GSD 记录中 40 篇 PDF 产生了大量 table artifacts，为后续 table chunks 和 facts 提供来源。

---

## 7.2 Phase 15：Table Chunking And Structured Facts

### Decide / Action

在 backend corpus import 阶段，我们把 table artifacts 引入 processed corpus：

- 增加 chunk 类型：
  - `text`
  - `table`
  - `table_row`
  - `table_summary`
- 生成 table chunks。
- 生成 table row chunks。
- 抽取核心财务事实，形成 `table_facts.json`。
- 对关键字段做归一化：
  - metric。
  - period。
  - raw value。
  - unit。
  - currency。
  - table id。
  - source document。
- 重建 corpus 和 index。

### Result

系统从“表格可保存”进一步变成“表格可检索、可引用、可结构化使用”。

阶段结果包括：

- 9303 text chunks。
- 4128 table chunks。
- 660 table_row chunks。
- 1558 structured facts。
- 合计 14091 chunks/vectors。

这使系统能够围绕财务指标构建比纯文本更强的证据层。

---

## 7.3 Phase 16：Table-Aware Retrieval And QA Integration

### Observe

有了表格 facts 后，问题变成：如何让问答真的优先使用这些结构化事实，而不是仍然只靠普通向量检索？

尤其是类似：

- “英伟达 FY2026Q3 最近一个季度总营收是多少？”
- “NVIDIA 2026 年第三季度数据中心业务贡献多少？”
- “某公司最新营收是多少？”

这些问题对 period、metric、company 非常敏感。

### Orient

我们判断不能只增加 chunk 数量并依赖模型“自己理解”。因为：

- 向量召回可能召回语义接近但 period 错误的数据。
- Rerank 可能偏好含有关键词的 chunk，但不保证年份/季度正确。
- LLM 生成时可能混合多个期间的数据。
- “最新”是一个结构化排序问题，不是纯语义相似度问题。

### Decide / Action

Phase 16 增加了 table-aware retrieval 和 numeric QA 支撑：

- 对 numeric/metric 问题路由到 table facts。
- 检索时引入 metric / period / company 约束。
- Rerank 阶段提高 table fact 证据权重。
- citation metadata 暴露 table 信息：
  - table id。
  - metric。
  - period。
  - raw value。
  - unit。
  - currency。
- 针对财务数字型问题增加回归测试。
- 优化 query handling，让“最新季度”“FY2026Q3”等表达能映射到正确期间。

### Result

系统能回答更精确的财务表格问题。例如针对 NVIDIA FY2026 Q3 revenue，系统能稳定返回：

- `57,006`。
- 并给出 table fact citation。

这不是简单“多塞文本块”实现的，而是增加了结构化 facts 与 period-aware 过滤/排序。

# 8. 第八阶段：“最新数据”错误问题的专项修复

## Observe

你发现一个严重问题：原先能回答的“英伟达 2026 年第三季度总营收是多少”后来答不上或答错。

这类问题本质上是最新财务数据查询，风险点包括：

- FY 年份和自然年份混淆。
- Q3 / third quarter / 最近一个季度表达混淆。
- 同一文档中有年度、季度、同比、百分比等多个数字。
- 表格中百分比值可能被误当金额。
- 检索结果可能命中旧 period。
- Rerank 可能把语义相近但 period 错误的项排到前面。

### Orient

我们判断这不是单纯“向量召回不够”的问题，而是金融事实检索中的约束缺失问题。  
因此使用了更通用的修复方向：

- 不只针对某个问题 hardcode。
- 建立 period compatibility。
- 建立 metric/value 类型过滤。
- 防止 percentage value 进入 revenue amount 候选。
- 对 requested year 做硬过滤。
- 对 latest/current period 做规则化识别。

### Decide / Action

根据 debug 记录，我们做了：

- 给 table fact retrieval 增加 percent raw value 过滤。
- 增加 requested-year 过滤。
- 改进 current-period 识别，更多依赖 header / period label，而不是文件名或简单列名。
- 在 hybrid/rerank 中加入 period compatibility guard。
- 防止 table_fact 因为 boost 过高压过不兼容 period 的证据。
- 增加测试覆盖：
  - table facts。
  - table fact retrieval。
  - query API。
  - hybrid retrieval。
  - rerank service。
- 用真实查询验证：
  - 2026 Q3 返回 57,006。
  - 2024 Q3 返回 35,082。

### Result

“最新营收数据”问题被修正为一个更通用的 numeric/time-aware retrieval 能力，而不是单点 patch。

它能泛化到类似问题：

- 同一指标不同期间。
- 同一公司不同报告期。
- 年度 vs 季度。
- 金额 vs 百分比。
- 最新 / current / recent period 表达。

当然它不是万能自然语言时间推理系统，但对财务表格中的核心 period/metric 查询已经明显增强。

# 9. 第九阶段：前端 Markdown 与 Citation 渲染问题修复

## Observe

你发现某个问题百分百触发前端显示异常：

- Markdown 没有正常渲染。
- citation HTML 直接显示成字符串。
- 没有自动换行。
- 出现横向滚动条。

具体问题是：

`结合台积电先进制程与英伟达数据中心需求，AI 算力链条对两家公司业绩可能产生什么影响？`

### Orient

第一轮判断是 citation 格式兼容不足，因为模型可能输出：

- `[1]`
- `<sup>[1]</sup>`
- `<span class="cite" data-id="2">2</span>`
- `<sup><span class="cite" data-id="2">2</span></sup>`
- `[citation_id not found]`

但后续 debug 发现更深层根因：该 query 的模型输出把整段 Markdown 包进了：

```markdown
...
```

ReactMarkdown 正确地把它当代码块处理，所以：

- `###` 被当普通文本。
- `<span class="cite"...>` 被当普通字符串。
- 长行按 code block 展示，不自动换行。

### Decide / Action

我们做了前端渲染防御：

- 在 `ChatArea` 渲染前解包整段 `markdown/md` fenced block。
- 对模型输出的转义 citation HTML 做归一化。
- 清理无效 citation placeholder。
- 增加 `min-w-0`、`break-words`。
- CSS 中给 Markdown、pre、code 增加兜底换行。

### Result

这个问题从“特定问题必现”变成可被前端兼容处理。  
这类修复的价值在于：真实大模型有时会不稳定地把 Markdown 包进代码块，前端需要做容错，而不是完全假设模型总按 prompt 输出。

# 10. 第十阶段：部署与数据迁移策略

## Observe

你准备在服务器部署系统，但遇到问题：

- 代码可以下载。
- 但是缺少数据不能运行。
- 不确定应该上传 PDF、向量库，还是重新索引。
- 模型配置不同可能导致已有向量索引不可用。

### Orient

我们判断部署时要区分三类文件：

1. 源文件：PDF / raw documents。
2. 处理产物：processed documents/chunks/table_facts。
3. 索引产物：FAISS / BM25 / metadata index。

如果服务器模型配置和本地一致，可以迁移 processed + index 以节省时间。  
如果 embedding provider/model/dimension 改了，就必须重建向量索引。

### Decide / Action

我们在 README 和部署说明里强调：

- `.env` 不提交，需要服务器单独配置。
- 如果模型配置一致，可以迁移本地索引产物。
- 如果 embedding 模型更换，必须重建索引。
- 大文件不要进入 git。
- 建议用 Docker 部署，但也保留直接运行方式。
- 写明 backend/frontend 启动方式。

### Result

系统具备了更清晰的部署路径：

- 开发环境可直接运行。
- 服务器环境可通过 Docker 或直接 Python/Node 启动。
- 数据迁移策略明确。
- 避免把大索引文件误提交到 git。

# 12. 关键困难与我们的判断方式

## 困难一：金融 PDF 的核心信息在表格，而不是正文

**观察**

普通文本 chunk 能召回报告，但对财务数字的定位不可靠。

**判断**

不能只靠“更多 chunk + 更强模型”。金融数字问题需要结构化 facts。

**处理**

引入 table extraction、table chunks、table facts。

**结果**

数字型财务查询准确性明显提升。

---

## 困难二：“最新”不是语义问题，而是时间排序与 period 匹配问题

**观察**

向量检索能找到 NVIDIA revenue，但可能不是用户问的季度。

**判断**

latest/current/FY/Q3 需要显式 period 逻辑，不应完全交给 LLM。

**处理**

增加 period compatibility、requested-year filter、percent filter、current-period recognition。

**结果**

FY2026 Q3 revenue 等问题能稳定返回正确数值。

---

## 困难三：上传导入会破坏已有知识库

**观察**

上传一个 PDF 后，文档数和 chunk 数被清零。

**判断**

导入流程存在覆盖风险，必须从数据治理层修复。

**处理**

空导入不覆盖、临时目录导入、成功后 merge、回归测试。

**结果**

知识库导入变成安全增量流程。

---

## 困难四：模型输出不可完全控

**观察**

特定问题中模型把 Markdown 包成代码块，导致前端渲染失败。

**判断**

Prompt 约束不够，前端需要容错 normalization。

**处理**

渲染前 unwrap markdown fence，归一化 citation，增强换行。

**结果**

前端对异常 Markdown 输出更健壮。

---

## 困难五：模型 provider 与索引强绑定

**观察**

embedding 模型从百炼切换到硅基流动后，旧索引不可继续假设可用。

**判断**

向量维度、embedding 分布、模型语义空间都可能变化，必须重建索引。

**处理**

支持 provider 配置并重建 bge-m3 索引。

**结果**

SiliconFlow embedding/rerank 可用，索引维度和 provider 匹配。
