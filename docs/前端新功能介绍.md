# 前端新功能介绍

# 系统检索链路模块说明

## 1. QUERY PLAN（来源：Phase 17 — 结构化查询理解）

基于`RetrievalPlan`对象生成，所有路由、过滤逻辑均以此为基准

| 字段 | 截图取值 | 含义 |
| --- | --- | --- |
| 第一个蓝色chip | reasoning | intent：查询意图，三选一 factual / analytical / reasoning。决定生成 prompt 模板 |
| 第二个chip | causal_analysis | task_type：更细粒度的任务类型，六选一 metric_lookup/causal_analysis/risk_analysis/trend_analysis/comparison/general_analysis。"如何变化"+"原因" 命中 causal |
| 第三个chip | research_report_analysis | retrieval_strategy：检索策略，四选一 table_fact_first / financial_report_first / research_report_analysis / general_hybrid |
| Entities | 宁德时代 | 通过 OntologyMatcher 从 COMPANY_ALIASES 命中的规范实体名（"CATL"、"300750" 都会归一到"宁德时代"） |
| Metrics | revenue, yoy_change | 从 METRIC_ALIASES 命中的度量。"营业收入" → revenue，"同比变化" → yoy_change |
| Doc Types | research_report, financial_report | preferred_doc_types：基于 strategy 推断的偏好文档类型。研报分析路径优先研报，财报兜底 |
| Time | year: 2024, quarter: , fiscal: false, relative: , raw: ... | QueryTimeRange 五个字段：年/季度/是否财年/相对时间(latest/recent/recent_years)/原始文本 |
| Expanded | 宁德时代, CATL, catl, 300750, revenue, ... | QueryRewriteEvent.expanded：实体别名 + 度量同义词扩展，喂给 BM25 召回 |
| Sub queries | ... 相关事实依据 / ... 对公司经营和行业变量的影响 | sub_queries：基于 intent 生成的子查询，给 reasoning 类查询额外的语义角度 |

## 2. ROUTE & FILTERS（来源：Phase 18 — 知识路由和元数据预过滤）

展示路由决策与元数据过滤执行结果

| 字段 | 截图取值 | 含义 |
| --- | --- | --- |
| Route | research_report_analysis | choose_route() 最终选定的路由，决定后续召回偏好 |
| Reason | task:causal_analysis | 路由理由：因任务类型是 causal_analysis 命中"研报分析路径"分支。其它可能值：entity_metric_lookup、entity_time_filing_lookup、plan_strategy、fallback |
| Filter Count | 28 -> 20 | 候选数：过滤前 28 → 过滤后 20。代表 BM25/向量/补充三路命中合并后被元数据过滤掉了 8 个 |
| 橙色徽章 | metadata_filters_relaxed_to_all | 过滤松弛标志：严格过滤后候选不足，自动放宽（去掉 chunk_type/year/quarter 约束）后才得到结果。这个原因来自 apply_metadata_filters() 的兜底分支 |
| company | 宁德时代 | 从 plan.entities 投影出的实体过滤约束 |
| doc_type | research_report / financial_report | 从 plan.preferred_doc_types 投影 |
| metric | revenue / yoy_change | 从 plan.metrics 投影 |
| year | 2024 | 从 plan.time_range.year 投影 |
| chunk_type | section / text / table / table_row / table_fact | 根据 retrieval_strategy 选定的允许 chunk 类型集合。research_report_analysis 路径下允许全部类型，包括 Phase 22 新增的 section |

## 3. CASCADE（来源：Phase 19 — 检索级联追踪 + Phase 22 下钻）

核心端到端追踪单元，单阶段对应`RetrievalCascadeStage`，展示输入输出数量与执行方法

| 阶段 | 截图取值 | 含义 |
| --- | --- | --- |
| query_plan | 1 -> 1 方法 research_report_analysis | Phase 17 产物：把 1 个原始查询变成 1 个结构化 plan。method 字段直接写入选定 route 便于一眼看清 |
| coarse_recall | 1 -> 68 方法 bm25+vector+supplemental | Phase 18 的实际召回：BM25 + 向量 + 补充信号（table_fact 等）合并，68 = bm25_count + vector_count + supplemental_count |
| metadata_filter | 28 -> 20 方法 metadata_post_recall_filter | Phase 18 的过滤：注意"28"是被过滤的候选基数（取三路最大值），"20"是过滤后保留。橙色 metadata_filters_relaxed_to_all 表明触发了松弛兜底 |
| hierarchy_drill_down | 58 -> 0 方法 parent_child_metadata | Phase 22 的层级下钻：从命中的 section/table 父 chunk 扩展出子 chunk。58 是父级候选总数，0 表示这次查询没有命中任何 section/table 父级（所以没下钻出子证据） |
| fusion | 58 -> 20 方法 rrf | Phase 18/19 的 RRF 融合：把所有候选用 Reciprocal Rank Fusion 算法重排，截取 top-20 |
| rerank | 21 -> 5 方法 rerank_service | Phase 16/19 的重排：用 reranker 模型对 fusion 结果打分，取 top-5（21 是去重后输入数，与 fusion 的 20 略不同因为可能有 fact 引入） |
| final_evidence | 5 -> 5 方法 evidence_pack_compression | Phase 20 的证据压缩：5 个 rerank 结果经过 EvidencePack 去重 + 字段保留 + 文本截断，这次没有重复所以 5 → 5。如果有重复会变成例如 5 → 3 |

> 设计要点：metadata_filter 在 coarse_recall 之后而不是之前——意图上是"pre-filter"，实现上是"post-recall filter"，所以 metadata 中带 applied_at: "post_recall" 显式标注。
> 

## 4. ITERATIVE STEPS（来源：Phase 21 — 迭代检索 demo 模式）

触发条件：task_type 属于 risk/causal/trend/comparison 且非 table_fact_first 路径，固定三步执行

| 步骤 | purpose | 内容含义 |
| --- | --- | --- |
| 1 | background_facts | "背景事实"：第一步检索查询 = 原始 query + anchors（实体+度量+时间） + 后缀"相关事实依据"。目的是先把"基础事实"召回出来 |
| 2 | risk_or_driver_evidence | "驱动/风险证据"：第二步根据 task_type 拼后缀：risk_analysis → "经营风险 政策变化 ..."、causal_analysis → "原因 影响 传导机制 驱动因素"、trend_analysis → "趋势变化 ..."。目的是召回"为什么/驱动力"的证据 |
| 3 | cross_check | "交叉验证"：仅 comparison/causal/risk 启用第三步，加后缀"交叉验证 对比 反向证据"。目的是找反向/对比证据避免单边视角 |

单步骤展示内容：

- 检索查询全文：实际拼好喂给 retriever 的字符串
- Route：该步骤命中的路由（通常和 plan 一致）
- Evidence chip 列表：该步骤前 5 个 chunk_id（截图都是 fact-xxx 是因为命中了 table_fact 类型）

> 补充说明：step 1/2/3 的查询文本基本一致只是后缀不同——这是 Phase 21 设计上的确定性规划（rule-based），便于复现和测试。生产化时可以替换为 LLM 驱动的真·agent 规划。
> 

## 5. HIERARCHY & DRILL-DOWN（来源：Phase 22 — 层级分块 & 下钻）

提取级联链路中下钻阶段详情

| 字段 | 截图取值 | 含义 |
| --- | --- | --- |
| hierarchy_drill_down | 阶段名 | 与 cascade 中同名 stage 对应 |
| 58 parents/candidates -> 0 child evidence | 输入 58 个父级候选，输出 0 个子证据 | "58" 是候选总数（即 bm25+vector+supplemental 合并去重后）；只有 chunk_type ∈ {section, table} 的命中才算"父级"。这次 0 表示没有命中任何 section/table 父级——查询直接命中的是 table_fact 而非 section/table，所以下钻没启动 |

### 下钻逻辑

1. 扫描召回结果，挑出 chunk_type ∈ {section, table} 的命中 → 视为"父级"
2. 通过 parent_id → children 反向索引（懒构建+缓存）拿到子 chunk
3. 跳过已经在候选集中的子 chunk（避免重复）
4. 给每个子 chunk 固定分数 HIERARCHY_DRILL_DOWN_SCORE = 0.12，进入 supplemental_hits 参与后续 RRF 融合
5. 限制规则：每父最多 3 个子，全局最多 8 个

## 完整查询链路示例

示例查询：宁德时代 2024 年营业收入是多少？同比变化如何？

```
原始查询
   │
   ▼
[QUERY PLAN]                    Phase 17
intent=reasoning, task=causal_analysis
strategy=research_report_analysis
entities=[宁德时代], metrics=[revenue, yoy_change]
   │
   ▼
[ROUTE & FILTERS]               Phase 18
choose_route → research_report_analysis
build_metadata_filters → {company, doc_type, metric, year, chunk_type}
触发松弛 → metadata_filters_relaxed_to_all
   │
   ▼
[CASCADE 1-2] query_plan → coarse_recall   1→68
   │
   ▼
[CASCADE 3] metadata_filter   28→20
   │
   ▼
[CASCADE 4] hierarchy_drill_down   58→0    Phase 22
（无 section/table 父级命中，下钻空跑）
   │
   ▼
[ITERATIVE STEPS] 3 步执行     Phase 21
每步独立调用 retriever.retrieve()，结果合并去重
   │
   ▼
[CASCADE 5] fusion (RRF)   58→20
   │
   ▼
[CASCADE 6] rerank   21→5
   │
   ▼
[CASCADE 7] final_evidence   5→5          Phase 20
EvidencePack 去重 + 字段保留 + 文本截断
   │
   ▼
生成 prompt → LLM → answer
```

## 面板与后端字段映射

| 前端面板 | 后端数据来源 | Pydantic 模型 |
| --- | --- | --- |
| QUERY PLAN | query_rewrite SSE 事件 .plan | RetrievalPlan |
| ROUTE & FILTERS | retrieval_complete.cascade_trace 中 query_plan + metadata_filter 阶段的 metadata | RetrievalCascadeStage |
| CASCADE | retrieval_complete.cascade_trace + rerank_complete.cascade_trace | RetrievalCascadeStage[] |
| ITERATIVE STEPS | retrieval_complete.iterative_trace.steps | IterativeRetrievalStep[] |
| HIERARCHY & DRILL-DOWN | cascade_trace 中 name=hierarchy_drill_down 阶段 | RetrievalCascadeStage |

调试接口：直接 POST /api/debug/retrieval，请求体`{"query": "..."}`，单次接口返回可囊括全部字段，前端面板为该数据可视化分组展示。
