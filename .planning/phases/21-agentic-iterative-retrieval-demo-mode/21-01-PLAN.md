<threat_model>
<assets>
- Financial source-grounded answers and citation metadata.
- Retrieval/debug traces that demonstrate how evidence was selected.
- Existing table-fact numeric QA correctness.
- API/SSE compatibility for frontend consumers.
</assets>

<threats>
- **Medium: Misleading trace data** if iterative steps record route/filter/evidence inconsistently with actual retrieval outputs.
  Mitigation: derive trace fields directly from `HybridRetrievalResult` and selected result items; cover with tests.
- **Medium: Evidence dilution** if merged multi-step candidates duplicate or bury high-quality table/text evidence.
  Mitigation: deterministic dedupe before rerank and final generation through existing evidence pack.
- **Medium: Factual QA regression** if simple numeric lookups trigger iterative retrieval.
  Mitigation: strict task-type/strategy eligibility tests and table-fact regression tests.
- **Low: Information leakage through debug traces** because debug responses expose retrieved snippets and metadata already visible in retrieval payloads.
  Mitigation: no new document access path; preserve existing debug endpoint behavior and avoid adding raw provider secrets or env data.
</threats>

<security_acceptance>
- Iterative trace never includes API keys, environment variables, or provider request payloads.
- Fallback paths still return normal bounded retrieval/rerank payloads.
- Debug/API payloads expose only retrieved evidence metadata already available through existing retrieval results.
</security_acceptance>
</threat_model>
