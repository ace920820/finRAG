<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> QueryRequest.query | Untrusted user text enters FastAPI and Pydantic validation before parser logic. |
| QueryRequest.query -> query parser | Arbitrary Unicode, punctuation, and long strings cross into deterministic normalization, regex, and alias matching. |
| retrieval plan -> future retrieval routing | Structured plan fields become metadata for later phases but must not be executed or used for filtering in Phase 17. |
| test/runtime -> provider clients | Phase 17 must not introduce network calls, LLM calls, model training, or service dependencies. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-17-01 | D | `normalize_query_text`, time regexes, alias matching | mitigate | Use bounded stdlib normalization and simple regex patterns only; add parser tests for full-width punctuation, Chinese punctuation, and required fiscal-quarter formats; avoid nested catastrophic regexes. |
| T-17-02 | T | `RetrievalPlan.filters` and `signals` | mitigate | Treat plan fields as inert extracted metadata in Phase 17; do not pass filters to `HybridRetriever.retrieve()` or execute values in workflow code. |
| T-17-03 | I | provider/test runtime | mitigate | Do not add LLM, embedding, rerank, Duckling, dateparser service, or network calls; keep tests on existing mock provider fixtures. |
| T-17-04 | D | malformed Unicode or empty/whitespace queries | mitigate | Preserve Pydantic `QueryRequest.query` min_length validation and make parser functions handle stripped strings without raising on unusual punctuation. |
| T-17-05 | R | parser explainability | mitigate | Populate deterministic `signals` strings such as `entity:NVIDIA` and `strategy:table_fact_first` so parser decisions are test-visible. |
</threat_model>
