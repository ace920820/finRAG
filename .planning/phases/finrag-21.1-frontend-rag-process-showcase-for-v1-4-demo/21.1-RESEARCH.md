# Phase 21.1: Frontend RAG Process Showcase For v1.4 Demo - Research

**Researched:** 2026-05-21
**Phase:** 21.1 - Frontend RAG Process Showcase For v1.4 Demo

## Research Goal

Plan frontend changes that expose the already-implemented v1.4 RAG intermediate artifacts from Phases 17-21 without fabricating Phase 22 hierarchy/drill-down data.

## Current Frontend Shape

### Data Flow

- `frontend/src/api/finrag.ts` parses `/api/query` SSE events.
- The frontend currently handles:
  - `query_rewrite`
  - `intent_detected`
  - `retrieval_complete`
  - `rerank_complete`
  - `answer_chunk`
  - `done`
- Current TypeScript payload types do not include:
  - `query_rewrite.plan`
  - retrieval `cascade_trace`
  - retrieval `iterative_trace`
  - rerank `cascade_trace`
  - richer citation metadata

### State Shape

- `frontend/src/types.ts` has `RetrievalSnapshot` with:
  - `bm25Docs`
  - `vectorDocs`
  - `rerankDocs`
  - channel errors
  - citations
- It does not store plan, route/filter/cascade, or iterative metadata.

### Display Shape

- `frontend/src/App.tsx` stores snapshots per assistant turn.
- `frontend/src/components/SidebarRight.tsx` displays three panels:
  - BM25 keyword recall
  - Vector semantic recall
  - Rerank top evidence
- `frontend/src/components/ChatArea.tsx` shows static stage labels with hard-coded durations/counts, not actual pipeline trace data.

## Backend Data Available Now

### Query Plan

`query_rewrite` includes `plan`, with fields from Phase 17:

- `original_query`
- `normalized_query`
- `intent`
- `task_type`
- `entities`
- `metrics`
- `time_range`
- `preferred_doc_types`
- `retrieval_strategy`
- `filters`
- `signals`

### Retrieval Payload

`retrieval_complete` includes:

- `bm25_results`
- `vector_results`
- `fused_top20`
- `bm25_error`
- `vector_error`
- `cascade_trace`
- `iterative_trace`

The route/filter metadata is present in `cascade_trace` stage metadata and in each iterative step's route/filter fields.

### Rerank Payload

`rerank_complete` includes:

- `top5`
- degraded/fallback metadata
- `score_source`
- rerank/final-evidence `cascade_trace`

### Done Payload

`done.citations` carries citation metadata used by clickable citations. Some table-fact metadata is nested under citation `metadata` from backend, but the frontend type currently omits that field.

## Recommended Implementation Path

### 1. Extend Frontend Types And Snapshot State

Add frontend TypeScript interfaces for:

- `RetrievalPlan`
- `RetrievalCascadeStage`
- `IterativeRetrievalTrace`
- `IterativeRetrievalStep`
- richer `CitationMetadata.metadata`

Extend `RetrievalSnapshot` with:

- `plan`
- `intent`
- `template`
- `expanded`
- `subQueries`
- `retrievalCascadeTrace`
- `rerankCascadeTrace`
- `iterativeTrace`

Keep all additions optional so older/mock snapshots remain valid.

### 2. Capture Live SSE Metadata

In `App.tsx`:

- store plan/expanded/subqueries on `query_rewrite`
- store intent/template on `intent_detected`
- store retrieval cascade and iterative trace on `retrieval_complete`
- store rerank cascade on `rerank_complete`
- keep existing BM25/vector/rerank mapping unchanged

Do not add new backend events.

### 3. Add A Process Inspector In The Right Sidebar

Refactor or extend `SidebarRight.tsx` to show:

- Query Plan section
- Route & Filters section
- Cascade section
- Iterative Steps section
- Evidence panels
- disabled Hierarchy Preview section

Recommendation: keep existing BM25/Vector/Rerank panels, but put them under or beside the new process sections. A tabbed view or accordion stack is acceptable. Given the existing sidebar, accordion stack is lower risk.

### 4. Keep Demo-Oriented But Honest

Do:

- show real backend values;
- show unavailable states clearly;
- show single-pass behavior when `iterative_trace` is missing;
- include Phase 22 preview as disabled/empty.

Do not:

- mock hierarchy parent/child data;
- invent cascade stages;
- replace existing chat/citation behavior;
- add new backend routes unless absolutely necessary.

## Testing Strategy

Frontend verification:

```bash
cd frontend && npm run lint
cd frontend && npm run build
```

If test framework is absent, use TypeScript/build checks as the minimum. Add focused pure helper tests only if a local test harness already exists.

Manual/demo verification after implementation:

- `宁德时代近期有哪些潜在经营风险？`
  - shows query plan and iterative steps.
  - shows route/filter and cascade counts.
- `宏观消费变化如何影响贵州茅台盈利能力？`
  - shows reasoning/causal plan and iterative steps.
- `英伟达2026年第三季度的总营收是多少？`
  - shows table-oriented single-pass behavior.
  - does not show fake iterative or hierarchy data.

## Risks And Mitigations

- **Overloading the sidebar:** use expandable sections and compact badges.
- **Type drift from backend:** define frontend types from current backend payloads and keep fields optional.
- **Fake demo risk:** show only real data, with explicit empty/disabled Phase 22 preview.
- **Citation regression:** preserve existing `RerankItem` IDs and citation click path.
- **Mobile/width overflow:** wrap long generated queries, filters, and evidence titles; avoid fixed-width text containers inside the sidebar.

## Recommendation

Implement as one frontend-focused plan:

1. Extend types and snapshot capture.
2. Add process inspector components in the right sidebar.
3. Preserve evidence/citation panels and add honest hierarchy preview empty state.
4. Validate with TypeScript/build checks and representative manual queries.

## Validation Architecture

This phase is validated by frontend type/build checks plus manual demo criteria. No backend reindex or corpus processing should be required.
