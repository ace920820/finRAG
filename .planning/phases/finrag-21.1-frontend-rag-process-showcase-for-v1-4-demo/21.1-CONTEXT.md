# Phase 21.1: Frontend RAG Process Showcase For v1.4 Demo - Context

**Gathered:** 2026-05-21
**Status:** Ready for planning
**Source:** User request during Phase 21 post-execution review

<domain>
## Phase Boundary

Phase 21.1 adds frontend display and interaction surfaces that make the v1.4 advanced RAG pipeline visible and compelling for an interview/demo environment.

This is not a production-minimal UX phase. The system's goal is to demonstrate engineering capability and expose intermediate processing steps clearly: query understanding, routing, metadata filtering, cascade trace, evidence compression/final evidence, and iterative retrieval steps.

This phase should happen before Phase 22 hierarchy/drill-down work so the user can try and present the existing Phase 17-21 backend capabilities immediately.

</domain>

<decisions>
## Implementation Decisions

### Demo Objective

- The frontend should prioritize explaining the RAG process over hiding complexity.
- Intermediate artifacts are first-class demo content, not internal debug noise.
- The UI should help a viewer understand why an answer was produced, what evidence path was used, and where fallback/degradation occurred.

### Content To Display

- Query understanding:
  - original query
  - expanded terms
  - sub-queries
  - intent
  - task type
  - entities
  - metrics
  - time range
  - preferred document types
  - retrieval strategy
- Routing and filtering:
  - route choice
  - route reason if available
  - applied metadata filters
  - candidate counts before/after filtering
  - relaxed filters or fallback reason
- Retrieval cascade:
  - stage names
  - method/source
  - input/output counts
  - degraded/fallback metadata
  - clear relationship between query plan, coarse recall, metadata filter, fusion, rerank, and final evidence
- Iterative retrieval:
  - step purpose
  - generated retrieval query
  - route
  - applied filters
  - selected evidence IDs and short evidence summaries
  - fallback/degraded reason when applicable
- Evidence/citations:
  - final top evidence remains visible and tied to clickable citations
  - table fact metadata should remain inspectable for numeric QA

### Interaction Direction

- Keep the chat flow as the primary surface.
- Enhance the right-side retrieval panel rather than replacing the whole app.
- Add compact but information-rich panels/tabs/sections for:
  - "Query Plan"
  - "Route & Filters"
  - "Cascade"
  - "Iterative Steps"
  - "Final Evidence"
- Analytical/reasoning examples should visibly show iterative steps.
- Simple factual/table QA should visibly show that it stayed single-pass when appropriate.

### Technical Constraints

- Reuse existing `/api/query` SSE payloads where possible.
- Do not add new SSE event names unless planning discovers that an existing backend field is unavailable through the current stream.
- Preserve existing answer streaming, citation click behavior, KB navigation, and document open behavior.
- Keep frontend changes scoped to display and interaction; do not rework retrieval/indexing/chunking in this phase.
- No corpus reimport or index rebuild should be required by this frontend showcase phase.

</decisions>

<specifics>
## Specific Ideas

- Extend `frontend/src/api/finrag.ts` types to include:
  - `QueryRewritePayload.plan`
  - retrieval/rerank `cascade_trace`
  - retrieval `iterative_trace`
  - citation metadata fields already present from backend
- Extend `RetrievalSnapshot` to store plan, trace, and iterative metadata per assistant turn.
- Add a demo-oriented trace component in the right sidebar with expandable sections.
- Use badges/chips for task type, route, strategy, doc types, filters, and fallback status.
- Use a vertical timeline or ordered list for cascade stages and iterative retrieval steps.
- Keep the existing BM25/Vector/Rerank panels as evidence detail sections.
- Validate with representative queries:
  - `宁德时代近期有哪些潜在经营风险？` should show iterative steps.
  - `宏观消费变化如何影响贵州茅台盈利能力？` should show reasoning/causal planning and iterative steps.
  - `英伟达2026年第三季度的总营收是多少？` should remain single-pass/table-fact oriented.

</specifics>

<deferred>
## Deferred Ideas

- Full frontend redesign or landing-page style presentation.
- New backend event names unless absolutely necessary.
- Rebuilding chunks/indexes.
- Phase 22 hierarchy/drill-down visualization until hierarchy metadata exists.
- Graph memory visualization.

</deferred>
