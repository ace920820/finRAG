# Phase 21.1 UI-SPEC — Frontend RAG Process Showcase

**Created:** 2026-05-21
**Phase:** 21.1 - Frontend RAG Process Showcase For v1.4 Demo
**Status:** Design contract ready for planning

## Intent

Make the v1.4 RAG pipeline visible inside the existing chat application for interview/demo walkthroughs.

This phase should look like a dense research/debug workbench, not a marketing page. The viewer should be able to follow how a question became a retrieval plan, how retrieval was routed and filtered, what stages ran, which iterative steps happened, and which final evidence supported the answer.

## Audience

- Interviewers evaluating backend/RAG engineering depth.
- Financial research users who need trust and source traceability.
- Demo presenters who need to explain intermediate steps quickly.

## Information Architecture

Keep the existing three-column app:

- Left: document library and example selection.
- Center: chat and answer stream.
- Right: expanded RAG process inspector.

The right sidebar becomes the primary showcase surface. It should support compact sections or tabs for:

- Query Plan
- Route & Filters
- Cascade
- Iterative Steps
- Final Evidence
- Hierarchy Preview

## Required Display Content

### Query Plan

Show only real backend fields from `query_rewrite.plan`:

- intent
- task type
- retrieval strategy
- entities
- metrics
- time range
- preferred document types
- expanded terms
- sub-queries

Use compact badges/chips and small key-value rows.

### Route & Filters

Show real route/filter data from retrieval payload and cascade metadata:

- route
- route reason when available
- applied filters
- filter input/output counts
- relaxed/degraded status
- fallback reason

Filters should render as compact chips or key-value rows, not raw JSON by default. A raw metadata fallback can be available in small text for unusual fields.

### Cascade

Show retrieval/rerank stages from `cascade_trace`:

- stage name
- method
- input count
- output count
- degraded flag
- fallback reason

Use a vertical timeline or ordered step list. Counts should be visually easy to scan.

### Iterative Steps

Show real `iterative_trace` when present:

- enabled/degraded/fallback status
- step purpose
- generated retrieval query
- route
- applied filters
- selected evidence IDs
- selected evidence summaries

If absent for factual/table queries, show a concise single-pass state rather than an empty error.

### Final Evidence

Keep the existing rerank evidence list and citation click behavior. Enhance only enough to expose:

- score source
- degraded/fallback reason
- table fact metadata when present through citations or evidence metadata

### Hierarchy Preview

Phase 22 is not implemented yet. Add only a disabled or empty-state preview section:

- label: hierarchy/drill-down planned
- no fake hierarchy data
- no mock parent/child relationships

## Interaction Requirements

- Preserve current answer streaming.
- Preserve citation click behavior and active evidence highlighting.
- Preserve KB navigation and document open behavior.
- Right sidebar sections should be expandable/collapsible or tabbed.
- Analytical/reasoning queries should automatically make iterative/cascade content easy to find.
- Factual/table QA should clearly show single-pass behavior.

## Visual Contract

- Dense operational UI.
- No landing page.
- No hero, decorative gradients, or marketing composition.
- No nested cards inside cards.
- Cards are acceptable only for repeated evidence/step items.
- Use restrained colors with multiple status accents, not a one-note blue/purple palette.
- Use small, legible labels and stable widths.
- Avoid viewport-scaled typography.
- Ensure long query strings, filters, and evidence titles wrap without overflow.
- Keep right-sidebar content scannable at 340-420px width.

## Data Policy

- Use real backend payloads from `/api/query`.
- Do not fabricate Phase 22 hierarchy/drill-down data.
- Mock/sample data may remain only for the app's existing no-query default state, but must be visually distinguishable from live assistant-turn snapshots.
- If a backend field is missing, render a clear unavailable/empty state.

## Verification Expectations

- `npm run lint` passes.
- Existing chat interaction still streams answers.
- Representative query `宁德时代近期有哪些潜在经营风险？` shows iterative steps.
- Representative query `英伟达2026年第三季度的总营收是多少？` shows single-pass/table-fact behavior and no fake iterative steps.
- Right sidebar does not overflow with long filters, generated queries, or evidence titles.
