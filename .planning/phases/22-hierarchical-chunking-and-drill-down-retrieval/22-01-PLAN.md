## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| raw corpus files -> processed chunk JSON | Extracted Markdown/table artifacts become persisted metadata visible through KB/document APIs and retrieval citations. |
| processed chunks -> hybrid retrieval candidates | Imported metadata controls filtering, parent-child expansion, and final evidence candidates. |
| retrieval trace -> API/SSE/debug consumers | Cascade metadata is serialized to existing frontend/debug consumers, including Phase 21.1 display surfaces downstream. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-22-01 | I/T | `Chunk.metadata` hierarchy fields and KB/document serialization | mitigate | Keep hierarchy additive under `metadata`; do not add required top-level `Chunk` fields. Store stable section titles/paths and normalized source-relative identifiers only. Add tests that old minimal `Chunk(**item)` payloads still parse and generated chunks do not require new top-level fields. |
| T-22-02 | T/I | `HybridRetriever` child expansion before fusion/rerank | mitigate | Gate drill-down by plan strategy/task type, dedupe by chunk ID, preserve table-fact strict-period filtering, and add a regression proving NVIDIA table-fact numeric QA still retrieves table facts after drill-down support. |
| T-22-03 | D | parent-to-child expansion over `bm25_store.chunks` | mitigate | Bound children per parent and total drill-down children, record input/output counts in `hierarchy_drill_down`, and skip expansion when no eligible parent chunks exist. Block completion if tests show unbounded candidate growth or missing caps. |
</threat_model>
