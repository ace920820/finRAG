# Requirements: FinRAG v1.1 Document Import Pipeline

**Defined:** 2026-05-13  
**Milestone:** v1.1 Document Import Pipeline
**Goal:** Replace hand-maintained fixture data with a repeatable PDF/Markdown document import pipeline that can feed FinRAG documents, chunks, and retrieval indexes.

## Context

v1.0 delivered the mock-data RAG MVP, including backend retrieval/rerank/generation, frontend-backend integration, and query rewrite preview. The remaining gap is real data ingestion: current reference data lives in `backend/app/data/processed/*.json` and was created from demo fixtures.

The user provided an existing PDF extraction project at `pdf2md/` that converts PDFs into structured Markdown plus manifests. v1.1 should reuse or adapt that code where practical, but reshape outputs for FinRAG's `Document` and `Chunk` schemas.

## v1.1 Requirements

### PDF Extraction Adapter

- [x] **PDF-01**: `pdf2md` can be invoked for FinRAG raw PDFs without Elite Daily season-specific assumptions blocking generic financial documents.
- [x] **PDF-02**: Extracted Markdown preserves source traceability: source PDF path/name, title, page count, extraction status, and content hash.
- [x] **PDF-03**: PDF extraction is idempotent: unchanged PDFs are skipped unless forced.
- [x] **PDF-04**: Extraction failures are recorded per file and do not abort the full batch.
- [x] **PDF-05**: The adapter can write outputs into a FinRAG-owned raw/extracted directory, not only an Obsidian Elite Daily vault layout.

### FinRAG Import And Chunking

- [x] **ING-01**: FinRAG has a documented raw data location for PDFs/Markdown inputs, such as `backend/app/data/raw/`.
- [x] **ING-02**: Import script converts extracted Markdown or plain text into `documents.json` and `chunks.json` using existing backend schemas.
- [x] **ING-03**: Chunking preserves document title, source, date/company/doc type metadata when available.
- [x] **ING-04**: Chunk IDs and document IDs are deterministic across repeated imports when source content is unchanged.
- [x] **ING-05**: Import pipeline supports at least PDF-derived Markdown and manually supplied Markdown/text files.
- [x] **ING-06**: Import pipeline can rebuild retrieval indexes after import.
- [x] **ING-07**: Existing demo fixture path remains available as a fallback and tests remain deterministic.
- [x] **ING-08**: Import pipeline has tests covering a small sample document and validating generated document/chunk JSON shapes.

### Integration And Operations

- [x] **PIPE-01**: A single documented command or small command sequence runs extraction → import → index build.
- [x] **PIPE-02**: README or planning docs explain how to add new source PDFs and rebuild the corpus.
- [x] **PIPE-03**: The frontend document library reflects imported documents through existing `GET /api/documents` without frontend redesign.
- [x] **PIPE-04**: Backend query flow can retrieve from imported chunks after index rebuild.

## Out Of Scope

| Feature | Reason |
| --- | --- |
| OCR for scanned PDFs | Existing `pdf2md` explicitly avoids OCR; v1.1 focuses on text-layer PDFs. |
| Production ingestion UI | Local CLI/script pipeline is enough for the current project. |
| Large-scale incremental indexing | Deterministic rebuild is sufficient before optimizing incremental updates. |
| Knowledge graph extraction | Future enhancement after reliable corpus ingestion. |
| Full financial table reconstruction | v1.1 should preserve text/table text as extracted; advanced table normalization can come later. |
