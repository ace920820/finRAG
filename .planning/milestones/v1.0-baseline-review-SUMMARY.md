## Findings By Severity

### Blocker

- **B1 â€” Real 40-PDF corpus lives only in `index/`, not in `processed/`.**
  - Files: `backend/app/data/processed/{documents,chunks}.json` (4 docs / 7 chunks demo seeds) vs `backend/app/data/index/{bm25_index,vector_index}.json` (40 docs / 9303 chunks).
  - APIs affected: `/api/documents`, `/api/kb/overview`, `/api/kb/documents`, `/api/kb/documents/{id}` all return the 4-doc demo view.
  - STATE.md (`kb-corpus-40-restore`) records "restored 40 documents/9303 chunks", but that state cannot be reproduced from the processed contract â€” only the cached indexes still carry it.
  - Owning phases: surfaces inside Phase 1 (`/api/documents`), but root cause is the v1.1 import pipeline (Phases 6-7) and v1.3 KB import write path (Phase 13).

- **B2 â€” Vector index dimension does not match live embedding provider.**
  - File: `backend/app/core/retrieval/vector_store.py:68-72`, `backend/app/data/index/vector_index.json`.
  - `vector_index.json` was built under `MockEmbeddingProvider` (dim=8). Local `.env` selects `FINRAG_EMBEDDING_PROVIDER=bailian` (dim=1024). Result: `vector_store.search` raises and `HybridRetriever.retrieve` discards BM25 too.
  - Reproduced end-to-end via TestClient on the NVIDIA Q3 revenue query: `bm25_results=[]`, `vector_results=[]`, `fused_top20=[]`.
  - Owning phase: Phase 2 retrieval foundation.

### Important

- **I1 â€” `HybridRetriever.retrieve` is all-or-nothing.** (`backend/app/core/retrieval/hybrid.py:39-49`) Vector failure should not erase BM25 results. Affects fault tolerance.
- **I2 â€” No provider/dimension metadata on vector index.** (`backend/app/core/retrieval/vector_store.py`) Switching `FINRAG_EMBEDDING_PROVIDER` silently produces a broken index. No load-time check.
- **I3 â€” `query.py` reaches into `agent/workflow.py` private helpers.** (`backend/app/api/query.py:11`) Duplicates orchestration in two places.
- **I4 â€” `query.py` exception path can emit `rerank_complete` twice.** (`backend/app/api/query.py:65-72`) Minor SSE protocol noise.

### Nice-to-have

- **N1** Text providers live in `providers/rerank.py` rather than `providers/text.py`.
- **N2** `detect_entities` and `_expand_terms` maintain parallel alias lists.
- **N3** `PreviewRewriteResponse.expanded_terms` vs `QueryRewriteEvent.expanded` name drift.
- **N4** Hand-tuned NVIDIA-specific boosts in `rerank_service._query_alignment_boost` and `hybrid._supplemental_hits`. Should retire after Phase 16.
- **N5** Frontend bundle size warning at 674 kB.
- **N6** `vite.config.ts` still defines unused `process.env.GEMINI_API_KEY`.
- **N7** `mapRetrievalResults` `isHigh` threshold (0.85) rarely fires for RRF-fused scores.
- **N8** External-drive AppleDouble `._*` files break `compileall`. Cosmetic, environment-specific.

## Phase 15/16 Risk Notes

- Phase 15 will add `chunk_type` and structured table chunks. `Chunk.metadata` already supports additive fields, but **B1 (processed/index drift) must be resolved first** so the import pipeline can produce a consistent processed + index pair.
- Phase 16 will route numeric queries through table evidence. The hand-tuned NVIDIA boosts (N4) will likely conflict with structured routing and should be retired once table chunks land.
- Phase 16 surfaces table citations through `RerankResultItem` and `CitationMetadata`; current shapes are additive-friendly, but **I2 (no index provenance)** will hurt debugging if vector + table indexes get out of sync.
